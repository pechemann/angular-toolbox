LAF-less Components

LAF-less Components, for "Look-and-Feel-less" Components, provide a GUI API built on top of native HTML, to enable common user actions independently from the application Look and Feel.

Motivation

A major recurring problem that developers face while developing Angular applications is the choice of graphic design. You can either use custom CSS, CSS frameworks, libraries built on top of CSS frameworks, or component libraries...

None of these solutions are bad in themselves; the choice will be made based on the type of project. However, when a choice is made, Developers often have to deal with the same problem: add a functionality not covered by the chosen solution.
In this case, the approach is always the same: include a library that solves the problem.

Making the right choice from the start is crucial!

Let us take a concrete use case: using the Bootstrap framework to create a web site.
If you decide to use the framework as is, you will include additional javascript libraries (e.g. jQuery), which is not the best solution. If you decide to use additional framework built on Boostrap, you may have new issues:

  • include more code that needed (e.g. component core modules of Component frameworks)
  • icrease security vulnerabilities (e.g. additional 3rd party library dependencies)
  • icrease learning curve
  • add strong dependency to the framework
  • icrease bundle size

Another major problem is that it is difficult for framework developers to adapt to Angular upgrade cycle.

On the other hand, if you do not use a framework you will have to take into account some other difficulties:

  • reinvent the wheel for basic behaviours (e.g. dropdown menus, etc.)
  • icrease bug occurences (e.g. poor test coverage, etc.)
  • add poorly designed code

The last option is to use an independent component library (e.g. PrimeNG, NG-ZORRO, etc.), which strongly constrains the graphics of your project.

The LAF-less Components Solution

In view of the various issues discussed above, Angular Toolbox offers an additional alternative to provide Angular developers with a finer granularity of choices.

The idea behind this is to separate the behavior of common components from the graphics framework rendering. According to ATX principles, these containers do not use third-party libraries and have a maximum test coverage.
The key point is to use the native HTML API to ensure a better code quality.

Another important point is to keep the use of containers as simple as possible, while maintaining maximum flexibility. To achieve this, we respect architectural principles, such as the separation of concerns. According to this logic, there is no coupling between the "container" behavior and the business code of the application.
This is why we often use the term Container to refer to the components of the LAF-less API. A good illustration of this concept is the behavior of the ATX Border Layout which does not resize the contents of containers, as Java BorderLayout class does.

As mentioned before, this solution does not claim to be better than others. It is just an additional tool to allow developers to make the best choice according to their needs.

LAF-less API

The following table shows the list of all LAF-less components available with the ATX framework:

Component Description API Demo
Border Layout Arranges the components to fit in the five regions: east, west, north, south, and center.
Dropdown Container Creates popup containers that are displayed by clicking on a button.
Dialog API Creates modal popups that display custom Angular components.
